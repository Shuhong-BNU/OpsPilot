# OpsPilot

[English Version](D:/桌面/codex/260415-OpsPilot%203.0/README.en.md)

> 基于 RAG 与 MCP 的智能运维助手

OpsPilot 面向企业运维场景，目标不是把大模型简单接进聊天框，而是把问题识别、知识检索、工具调用、诊断编排、状态沉淀和指标观测串成一个更完整的 Agent 工程闭环。

## 项目定位

这个项目更适合在简历和面试里被描述成：

- 面向企业运维场景的 Agent 工程项目
- 基于意图识别分流的智能运维助手
- 结合 RAG、MCP、工作流编排和状态持久化的工程化实践

这不是一个“硬包装成生产系统”的项目。当前表述刻意保持克制：

- 不宣称已接入真实生产工单平台
- 不宣称已完整落地真实 Prometheus/MySQL 生产系统
- 不宣称是多智能体平台
- 不宣称存在未经验证的降本或提效数据

一句话总结：

OpsPilot 更强调“链路完整、结构清晰、边界诚实”，而不是“故事讲得很满”。

## 为什么它不是普通聊天项目

OpsPilot 的重点不在“模型能回答问题”，而在“系统如何决定走哪条链路，并把结果沉淀下来”。

当前系统主要提供 6 类能力：

- 意图识别分流：把请求分到 `smalltalk / simple_qa / knowledge_qa / aiops_diagnosis / unsupported`
- 混合检索问答：Milvus 稠密检索、SQLite FTS5 稀疏检索、RRF 融合、轻量 rerank
- AIOps 诊断流程：基于 `Plan-Execute-Replan` 的流式故障分析
- MCP 工具调用：让日志、监控类能力通过工具层接入主流程
- 持久化状态管理：会话、消息、工作流运行记录、工具调用日志统一落库
- 轻量可观测性：请求量、平均耗时、检索耗时、rerank 耗时、MCP 成功率、AIOps 总耗时

这一层的价值在于：

- 简单问题不必走重型链路
- 文档问题有独立的检索工程叙事
- 排障问题有单独的工作流编排叙事
- 所有关键过程都能被记录、回放和讲清楚

## 整体架构

OpsPilot 当前采用“前端交互层 + API 接口层 + 编排服务层 + 检索与工具层 + 持久化层”的结构。

### 1. 前端交互层

这一层负责把项目变成一个可演示、可登录、可查看历史会话的运维助手界面。

- 提供登录态管理
- 保存 JWT 到浏览器本地
- 从后端同步会话列表
- 支持快速对话、流式对话和 AIOps 诊断入口

### 2. API 接口层

这一层负责把对话、AIOps、上传、会话和监控能力暴露成清晰接口。

- `/api/auth/*` 处理登录和当前用户信息
- `/api/chat*` 处理对话、流式输出和单会话清理
- `/api/sessions*` 处理会话列表和详情
- `/api/aiops` 处理流式诊断
- `/api/upload` 处理文档接入和索引
- `/metrics` 处理结构化指标暴露

### 3. 编排服务层

这一层是项目最核心的“Agent 工程”部分。

- `chat_service` 负责意图识别、链路路由、消息落库
- `aiops_service` 负责 `Plan-Execute-Replan` 工作流
- `intent_service` 负责规则优先、LLM 兜底的意图识别

### 4. 检索与工具层

这一层负责把知识和外部能力组织成可调用的执行单元。

- `retrieval_service` 负责 dense recall、sparse recall、RRF 和 rerank
- `knowledge_tool` 负责把检索结果转成问答上下文
- `mcp_client` 负责统一管理 MCP 工具调用与重试

### 5. 持久化层

这一层负责让系统不再依赖纯运行期内存。

- `sessions` 保存会话元数据和 `thread_id`
- `messages` 保存用户和助手消息
- `workflow_runs` 保存知识问答/AIOps 工作流记录
- `tool_call_logs` 保存工具调用日志
- `document_chunks` 与 `document_chunks_fts` 支撑稀疏检索

这套分层的意义在于：

既能把“问答、检索、诊断、工具、状态”分开讲，又能把它们串成一个完整闭环。

## Query 主链路

对面试来说，最值得讲的是一次对话请求如何从入口走到最终回答。

当前主链路可以拆成 6 步：

### 1. 请求进入聊天接口

用户通过 `/api/chat` 或 `/api/chat_stream` 发起请求，系统先校验 JWT，再读取当前登录用户。

这一层先做权限校验，是为了让后续会话、AIOps、上传这些能力具备最小安全边界。

### 2. 进入意图识别层

`chat_service` 首先调用 `intent_service.classify()`。

这里采用的是：

- 规则优先
- 不确定时再走轻量 LLM 分类

这样做的原因是：

- 运维项目里很多请求模式是稳定的
- 简单规则可以降低无效流量
- 可以避免让所有请求都走重型 Agent 链路

### 3. 根据意图路由到不同执行链

路由结果固定为 5 类：

- `smalltalk`
- `simple_qa`
- `knowledge_qa`
- `aiops_diagnosis`
- `unsupported`

对应动作分别是：

- 闲聊和简单问答直接回答
- 文档问题走检索链路
- 排障问题走 AIOps 工作流
- 越界请求直接拒答

### 4. 如果是知识问答，进入混合检索链路

`knowledge_qa` 会进入 `retrieval_service.hybrid_search()`。

这一步内部又分成 4 个子步骤：

1. 用 Milvus 做稠密召回
2. 用 SQLite FTS5 做稀疏召回
3. 用 RRF 融合候选集
4. 用轻量 rerank 做最终排序

这样设计的目的，是让你在面试里能明确讲出：

- 为什么不只做向量检索
- 为什么要保留关键词匹配能力
- 为什么需要融合与重排

### 5. 如果是 AIOps 诊断，进入工作流链路

`aiops_diagnosis` 会进入 `aiops_service` 的 `Plan-Execute-Replan` 工作流。

这条链路的意义不是“让模型想一想”，而是把排障过程拆成清晰阶段：

- Planner 生成诊断计划
- Executor 执行步骤并调用工具
- Replanner 基于结果决定继续执行还是收敛输出

这样更容易把排障逻辑讲成“可编排流程”，而不是一段大 Prompt。

### 6. 把结果写回状态层并更新指标

请求结束后，系统会把：

- 会话元信息
- 用户消息
- 助手回复
- 工作流运行状态
- 工具调用日志
- 请求耗时指标

统一写入数据库或指标服务。

这一步很关键，因为它决定了项目不只是“当场回答一次”，而是能恢复、回放、审计和复盘。

一句话总结这个主链路：

OpsPilot 的核心不是单点回答能力，而是“先路由、再执行、再落库、再可观测”。

## 混合检索设计

混合检索是这个项目里最值得体现工程深度的部分之一。

当前实现遵循一条收敛路线：

- 稠密检索继续使用 Milvus
- 稀疏检索使用 SQLite FTS5
- 融合使用 RRF
- 重排使用轻量可解释的 overlap rerank

### 为什么要做混合检索

单独做向量检索的问题是：

- 对关键词、术语、报错字面量不够稳定
- 对运维文档里的固定指标名、告警名、组件名不一定最敏感

单独做关键词检索的问题是：

- 语义泛化能力弱
- 同义表达和自然语言问题不稳定

把两者结合起来，能更自然地解释检索系统为什么更可信。

### 当前检索 trace 能讲什么

当前 trace 会记录：

- dense 命中数
- sparse 命中数
- fusion 命中数
- rerank 命中数
- dense/sparse/rerank 各自耗时
- 最终来源文档

这让项目从“调用了一个向量库”升级成“能讲清召回工程”。

## 状态持久化与会话管理

当前版本已经不再把 `MemorySaver` 视为唯一状态来源，而是把服务端数据库作为主状态源。

这部分主要解决 4 个问题：

- 会话恢复
- 历史对话查看
- 工作流记录回看
- 工具调用可追踪

### 当前落库内容

- `sessions`：会话标题、更新时间、`thread_id`
- `messages`：用户/助手消息、意图、路由
- `workflow_runs`：执行类型、状态、开始结束时间、耗时
- `tool_call_logs`：工具名、成功失败、耗时、输入输出

### 为什么这层重要

很多 Demo 型项目的问题是：

- 刷新页面状态就丢
- 对话不能恢复
- 工作流执行过程不可追踪
- 工具失败没有审计信息

OpsPilot 的这一层，至少把这些问题压到了一个可解释、可回看的最小工程闭环里。

## 鉴权与安全边界

OpsPilot 没有做复杂权限系统，但已经补上了一个“像企业系统”的最小安全边界。

当前方案包括：

- JWT 登录态
- `viewer / operator / admin` 三种角色
- `AIOps` 与上传接口限制为 `operator / admin`

这样设计的意义不在于权限模型多复杂，而在于你能在面试里明确说出：

- 哪些接口是普通查看者可用
- 哪些接口属于高风险操作
- 为什么工具调用和诊断能力不应该完全开放

## 可观测性与测试

这部分的目标不是堆满监控系统，而是补一层“够讲、够证据化”的工程能力。

### 当前指标

系统已经能记录：

- 请求量
- HTTP 请求平均时延
- RAG 检索耗时
- rerank 耗时
- MCP 工具成功率和失败率
- AIOps 工作流总耗时

### 当前测试方向

已经补上的测试包括：

- 鉴权服务测试
- 意图识别规则测试
- 检索融合与重排测试
- API 权限边界测试

这些测试的意义不是追求覆盖率数字，而是证明关键链路不是随意拼起来的。

## 简历和面试怎么讲

更推荐你用下面这种说法：

### 中文版一句话

OpsPilot 是一个面向企业运维场景的 Agent 工程项目，通过意图识别分流、混合检索与 rerank、Plan-Execute-Replan 工作流、MCP 工具调用和持久化状态管理，构建了从问题识别到排障建议生成的完整闭环。

### 中文版展开版

- 项目先通过规则优先的意图识别，把闲聊、简单问答、知识问答和 AIOps 诊断拆到不同链路
- 在知识问答链路里，用 Milvus + SQLite FTS5 + RRF + rerank 实现混合检索
- 在排障链路里，用 Plan-Execute-Replan 驱动诊断计划生成、工具执行和结果收敛
- 在工程化层面，通过 JWT、角色控制、会话持久化、工具调用日志和指标暴露，把项目从 Demo 提升为更完整的 Agent 系统原型

## 当前边界

为了保证表述诚实，当前边界也需要说清楚。

- 监控和工单数据更适合表述为“近真实样本”或“仿真数据闭环”
- rerank 目前是轻量实现，不是重型 cross-encoder 服务
- MCP 工具层已接入，但具体数据真实性依赖本地或仿真服务
- 测试已覆盖关键链路，但还不是大规模生产级测试矩阵

一句话收尾：

OpsPilot 最适合被讲成“一个链路完整、架构清晰、工程化设计明确、但叙事保持克制的智能运维 Agent 项目”。
