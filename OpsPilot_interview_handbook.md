# OpsPilot 复习手册

> 基于当前代码实现整理的项目复习版手册，适合面试前快速过一遍并直接口述。

## 1. 项目定位

OpsPilot 是一个面向企业运维场景的 Agent 工程项目，核心目标不是做一个“会聊天的大模型页面”，而是把问题识别、知识检索、工具调用、工作流诊断、状态持久化和轻量可观测性串成完整闭环。

这套系统当前最值得强调的能力有：

- 意图识别分流
- 混合检索与 rerank
- `Plan-Execute-Replan` 诊断编排
- MCP 工具调用
- 会话与工作流持久化
- JWT 与角色权限边界
- 指标采集与基础测试

一句话记忆：

OpsPilot 更像“结构清晰、边界诚实的智能运维 Agent 原型”，而不是硬包装成生产系统的 Demo。

## 2. 整体架构

OpsPilot 当前采用五层结构，便于把系统拆开讲，也便于解释每层在整个链路里的职责。

### 2.1 前端交互层

这一层负责把系统做成可登录、可提问、可看历史会话的页面。

- 支持 JWT 登录态
- 本地保存 token
- 从后端同步会话列表
- 提供快速对话、流式对话、AIOps 诊断入口

### 2.2 API 接口层

这一层把系统能力暴露成清晰接口。

- `/api/auth/*`：登录和获取当前用户
- `/api/chat*`：对话、流式输出、清空单会话
- `/api/sessions*`：会话列表和详情
- `/api/aiops`：AIOps 流式诊断
- `/api/upload`：文档上传和索引
- `/metrics`：轻量指标暴露
- `/health`：健康检查

### 2.3 编排服务层

这一层是项目最核心的 Agent 工程层。

- `chat_service`：统一聊天主链路，负责意图识别、路由和消息落库
- `intent_service`：规则优先、LLM 兜底的意图分类器
- `aiops_service`：诊断工作流执行器

### 2.4 检索与工具层

这一层负责让系统既能查知识，也能调工具。

- `retrieval_service`：dense recall、sparse recall、RRF、rerank
- `knowledge_tool`：把检索结果组织成问答上下文
- `mcp_client`：MCP 工具加载、重试和调用日志

### 2.5 持久化层

这一层负责把系统从“运行期记忆”提升到“服务端主状态源”。

- `sessions`
- `messages`
- `workflow_runs`
- `tool_call_logs`
- `document_chunks`
- `document_chunks_fts`

一句话记忆：

前端负责交互，API 负责入口，编排层负责决策，检索和工具层负责执行能力，持久化层负责沉淀和回放。

## 3. Query 主链路

Query 是这个项目最值得讲清楚的核心链路，因为它串起了意图识别、检索、诊断和状态管理。

### 第 1 步：请求进入接口

用户调用 `/api/chat` 或 `/api/chat_stream`。

这一层先做 JWT 校验并解析当前用户角色，保证后续链路具备最小权限边界。

### 第 2 步：进入意图识别层

`chat_service` 首先调用 `intent_service.classify()`。

当前固定输出是：

- `smalltalk`
- `simple_qa`
- `knowledge_qa`
- `aiops_diagnosis`
- `unsupported`

当前策略是：

- 规则优先
- 不确定时再走轻量 LLM 分类

这样做的原因是：

- 运维场景请求模式相对稳定
- 简单规则可以显著减少无效重型调用
- 可以把简单请求和复杂请求拆开处理

### 第 3 步：按意图分流

分流后的动作是固定的。

- `smalltalk`：轻量直答
- `simple_qa`：直接回答，缩短链路
- `knowledge_qa`：进入检索增强问答
- `aiops_diagnosis`：进入诊断工作流
- `unsupported`：直接拒答

这一步的价值，是让系统具备“先判断，再执行”的结构，而不是所有请求都直接喂给一个大模型。

### 第 4 步：如果是知识问答，进入混合检索

`knowledge_qa` 会进入 `retrieval_service.hybrid_search()`。

这一条链路包含：

1. Milvus 稠密检索
2. SQLite FTS5 稀疏检索
3. RRF 候选融合
4. 轻量 rerank

系统还会输出 retrieval trace，包括：

- dense/sparse/fusion/rerank 命中数
- 每个阶段耗时
- 最终命中文档来源

一句话记忆：

这一层的重点不是“查到了文档”，而是“检索工程链路是可解释的”。

### 第 5 步：如果是诊断请求，进入 AIOps 工作流

`aiops_diagnosis` 会进入 `Plan-Execute-Replan` 工作流。

工作流拆成三个节点：

- Planner：制定诊断计划
- Executor：执行步骤、调用工具
- Replanner：根据执行结果决定继续、调整或收敛

这样做的好处是：

- 诊断流程可分阶段解释
- 工具调用和推理过程更容易落日志
- 面试时能把它讲成“编排设计”，而不是“一个很长的 Prompt”

### 第 6 步：落库与指标更新

无论最后走的是知识问答还是诊断链路，系统都会把关键结果写回状态层。

当前会记录：

- 会话元数据
- 用户消息和助手回复
- 工作流运行状态
- 工具调用日志
- 请求耗时指标

一句话总结 Query：

OpsPilot 的主链路可以概括成“先路由，再执行，再沉淀”。

## 4. 混合检索设计

混合检索是项目里最容易体现“不是简单调库”的工程点之一。

### 当前实现

- Dense recall：Milvus
- Sparse recall：SQLite FTS5
- Candidate merge：RRF
- Rerank：基于 query-token overlap 的轻量重排

### 为什么要混合

单做向量检索的问题是：

- 对报错字符串、告警名、指标名不够敏感
- 对运维术语的字面匹配不总是稳定

单做关键词检索的问题是：

- 语义泛化能力不足
- 对自然语言表达不够稳定

所以当前设计选择把两者融合，而不是押注单一召回方式。

### 这部分面试时怎么讲

推荐讲法：

- 我没有把检索设计成“只查 Milvus”
- 我给它补了稀疏检索和融合层
- 这样一方面能保住术语命中，另一方面能兼顾语义召回
- 再通过 rerank 做最后收敛，检索 trace 也能用于调试和讲解

## 5. 持久化状态与会话管理

这部分是把项目从“运行时内存 Demo”升级成“可恢复系统原型”的关键。

### 当前表设计

- `sessions`：会话标题、`thread_id`、更新时间、最近意图
- `messages`：用户/助手消息、意图、路由、时间
- `workflow_runs`：工作流类型、状态、开始结束时间、耗时
- `tool_call_logs`：工具名、成功失败、耗时、输入输出

### 这一层解决什么问题

- 支持会话恢复
- 支持历史对话查看
- 支持 AIOps 结果回看
- 支持单会话清空
- 支持工具调用追踪

一句话记忆：

前端的 `localStorage` 现在只是缓存，后端 SQLite 才是主状态源。

## 6. 鉴权与安全边界

OpsPilot 当前没有做复杂 RBAC，但做了一个像企业系统的最小权限边界。

### 当前方案

- 登录态：JWT
- 角色：`viewer / operator / admin`
- 高风险接口限制：
  - AIOps 诊断：`operator+`
  - 文档上传：`operator+`

### 为什么这层重要

因为一旦引入工具调用和诊断能力，就需要解释：

- 什么能力可以开放给普通查看者
- 什么能力应该限制到更高权限
- 为什么诊断和工具执行不应该完全匿名开放

这会让项目从“一个能跑的页面”更像“有边界意识的系统”。

## 7. 可观测性与测试

这部分的目标不是铺满一整套监控平台，而是给项目补上工程证据。

### 当前指标

系统已经能记录：

- 请求量
- HTTP 请求平均时延
- 检索耗时
- rerank 耗时
- MCP 工具成功率/失败率
- AIOps 工作流总耗时

### 当前测试

已补测试方向包括：

- 鉴权服务测试
- 意图识别规则测试
- 检索融合与重排测试
- API 权限边界测试

一句话记忆：

测试和可观测性不是为了“显得全”，而是为了证明关键链路真的能跑通。

## 8. 面试时最值得强调的工程点

这里建议记住 5 个最核心的点。

### 1. 意图识别分流

不是所有请求都走统一链路，而是先判断请求类型，再决定直答、RAG 还是 AIOps。

### 2. 混合检索不是单点向量搜索

当前已经具备 dense recall、sparse recall、RRF 和 rerank，可以讲成完整的检索工程链。

### 3. AIOps 是工作流，不是一个长 Prompt

通过 `Plan-Execute-Replan` 把诊断拆成计划、执行、重规划三个阶段，更适合讲 Agent 编排能力。

### 4. 状态不是纯运行期内存

会话、消息、工作流运行和工具调用都有服务端持久化，这让系统具备恢复和审计能力。

### 5. 安全边界和观测能力已经补上

虽然没有复杂权限系统，但已经有 JWT、角色边界和指标采集，项目更像完整系统原型。

## 9. 当前边界

为了保证表述诚实，当前边界也要记住。

- 数据来源更适合表述为近真实文档、告警样本和仿真闭环
- rerank 目前是轻量实现，不是重型独立模型服务
- MCP 工具已接入，但能力真实性依赖本地或仿真服务
- 当前测试覆盖关键链路，但不是生产级大规模测试矩阵

## 10. 最后一段口述总结

如果面试官让你用一段话总结这个项目，可以这样说：

OpsPilot 是一个面向企业运维场景的 Agent 工程项目。它通过规则优先的意图识别，把简单问答、知识问答和 AIOps 诊断拆到不同链路；在知识问答部分引入混合检索和 rerank；在诊断部分采用 Plan-Execute-Replan 工作流；同时又补上 JWT、角色权限、会话持久化、工具日志和指标采集，所以它比普通 Demo 更完整，但在对外表述上仍然保持诚实克制。
